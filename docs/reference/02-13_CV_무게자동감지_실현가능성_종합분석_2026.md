## 2. 광학 분석: "0.5x 광각으로 찍으면?"

### 거리별 OCR 가능성 (플레이트 "20" 글자 높이 8cm 기준)

OCR이 읽으려면 **최소 20-30 픽셀** 필요 (Google ML Kit 최소 16px).

| 촬영 설정 | 2m 거리 | 3m 거리 | 4m 거리 | OCR 가능? |
|-----------|---------|---------|---------|----------|
| **0.5x 울트라와이드 (4K 영상)** | ~35px | ~23px | ~17px | 2m: 간신히 / 3m+: 불가 |
| **0.5x 울트라와이드 (48MP 스냅)** | ~92px | ~62px | ~46px | 양호 |
| **1x 메인 (4K 영상)** | ~70px | ~47px | ~35px | 양호 |
| **1x 메인 (48MP 스냅)** | ~183px | ~122px | ~92px | 매우 좋음 |

**핵심 딜레마**: 전신을 잡으려면 카메라가 2-4m 떨어져야 하고, 무게 숫자를 읽으려면 가까워야 한다. 거리가 멀어질수록 OCR은 안 되고, 가까워지면 전신이 안 잡힘 — **이건 물리적 한계**.

### 배럴 왜곡 (0.5x 광각)

- 120도 FOV 렌즈의 배럴 왜곡이 BlazePose 정확도를 **70-80%** 수준으로 떨어뜨림 (보정 없이)
- 하지만 Android ISP / iOS ISP가 **하드웨어 레벨에서 자동 보정** (<1ms)
- 보정 후 가장자리 잔여 왜곡 <2%, 중앙부 거의 완벽
- **실현 가능성: 9/10** (이미 해결된 문제)

### 듀얼 카메라 동시 사용

- iOS: AVCaptureMultiCamSession (iPhone 11 Pro+ 지원, hardwareCost < 1.0 제약)
- Android: getConcurrentCameraIds() (Android 11+, 기기별 편차 큼)
- **배터리 40-60% 증가**, 5-10분 내 열 스로틀링 발생
- **실현 가능성: 5/10** — 기기 파편화, 전력 소비 문제로 메인 전략 부적합

### 디지털 줌 + 크롭 전략 (권장)

0.5x 전체 프레임으로 포즈 추적 + 무게 플레이트 영역만 크롭하여 OCR:
- 4K 영상에서 25% 크롭 → 1920x1080 (2MP)
- 48MP 스냅에서 25% 크롭 → 4000x3000 (12MP)
- **실현 가능성: 8/10** — 가장 실용적, 모든 폰 지원

## 3. 모바일 CV 파이프라인: "동시에 두 모델 돌리면?"

### 개별 모델 성능

| 모델 | CPU 추론 | GPU 추론 | 비고 |
|------|---------|---------|------|
| BlazePose Lite | ~29 FPS (Pixel 3) | ~49 FPS (GPU) | 33 keypoints |
| YOLO11-N | 56.1ms CPU | 1.5ms T4 GPU | 물체 감지 |
| **YOLO26-N (2026.01 출시)** | **38.9ms CPU (43% 빨라짐)** | 1.7ms T4 GPU | NMS-free, 2.4M 파라미터 |

### 실행 전략 비교

| 전략 | 실현가능성 | Pose FPS | 배터리 | 복잡도 |
|------|:-:|:-:|:-:|:-:|
| A. 매 프레임 동시 실행 | **3/10** | 15-20 | 매우 나쁨 | 매우 높음 |
| B. Pose연속 + 세팅시만 Weight | **7/10** | 30 | 중간 | 중간 |
| **C. Two-Stage 분리 (권장)** | **9/10** | 30+ | 좋음 | 낮음 |

**Two-Stage 파이프라인**:
```
[무게 설정 단계]              [운동 진행 단계]
YOLO+OCR 전력 투입  →   Pose-only 모드 (30fps+)
"20kg 감지 완료!"          "1개... 2개... 3개..."
```
각 단계에서 해당 모델에 100% 자원 투입 → 최적 성능·정확도·배터리.

### YOLO26 — 게임체인저

2026년 1월 출시된 YOLO26-N이 가장 큰 기술 변화:
- NMS-free 아키텍처로 CPU 추론 43% 가속
- **단일 모델에서 Pose + Object Detection + Segmentation** 지원
- 2.4M 파라미터, mAP 40.9%
- TFLite/CoreML 내보내기 지원

## 5. 다중 인물 장면 처리

- MediaPipe Pose는 **단일 인물 전용** (num_poses 파라미터 있지만 실질적으로 1명만 제대로 동작)
- 권장 해결책: YOLO26-N으로 사람 검출 + **ByteTrack** 트래커로 ID 유지 + 타겟 영역만 Pose 적용
- 더 간단한 대안: 앱에서 사용자가 자신을 **탭으로 선택**
- ByteTrack은 고정 카메라에 최적, ~1-2ms 추가 비용

## 10. 전략 권고안

### 핵심: "포즈 추적과 무게 감지를 분리하라"

성공한 기업들은 전부 통제된 환경(홈짐/전용 하드웨어)에서만 카메라를 쓰고, 상용 헬스장에서 폰 카메라를 쓴 서비스는 폐업했다.

### 권장 로드맵

```
Phase 2a (즉시 착수 가능) ─────────────────────────────
│
│  [포즈 추적] 현재 Phase 1 시스템 유지 (검증 완료)
│  [무게 입력] 현재 V2log 수동 입력 UX 유지 (10초 기록)
│  [NEW] "플레이트 등록" 기능:
│        → 처음 1회 헬스장 플레이트를 사진으로 등록 (가까이서, 1x 카메라)
│        → 색상/크기/형태 패턴 학습 (YOLO26-N + ML Kit OCR)
│        → 다음부터 바벨 사진 찍으면 총 무게 자동 계산
│
Phase 2b (검증 후) ────────────────────────────────────
│
│  Two-Stage 파이프라인:
│  1단계: 운동 시작 전 0.5x로 바벨 촬영 → YOLO26+OCR → 무게 감지
│  2단계: "운동 시작" → Pose-only 모드 (30fps)
│  → 세트 사이 자동 재스캔 (무게 변경 감지)
│
Phase 2c (B2B 확장) ───────────────────────────────────
│
│  헬스장 설치형 카메라/스마트 미러 시스템
│  → 사용자 셋업 0, 프라이버시 헬스장 관리
│  → 기구 IoT 센서와 연동 (무게 자동 감지)
│  → 스마트 피트니스 미러 시장 $36B (연 25% 성장)
```

### YOLO 모델 파라미터 권고

| 항목 | 권장값 | 이유 |
|------|--------|------|
| **모델** | YOLO26-N (nano) | CPU 39ms, 2.4M 파라미터, 모바일 최적 |
| **입력 크기** | 640x640 | 표준, 속도/정확도 밸런스 |
| **클래스** | plate_face, plate_side, weight_text | 3개 클래스로 시작 |
| **학습 데이터** | 최소 500장 (현재 공개 데이터 92장뿐) | 자체 수집 필수 |
| **증강** | 회전(0-360도), 밝기변화, 노이즈, 부분가림 | 헬스장 환경 반영 |
| **배포** | TFLite (Android) + CoreML (iOS) | 온디바이스 추론 |
| **OCR** | Google ML Kit Text Recognition v2 | 무료, 온디바이스, 최소 16px |
