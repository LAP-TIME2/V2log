## 2-1. 창업아이템 개발/개선 준비현황

### (1) 기존 서비스 'V2log' 앱 개발 완료 현황

피카(Fica)의 모체 서비스인 **V2log**는 현재 Flutter 기반 크로스플랫폼 앱으로 **MVP 개발이 완료**된 상태이며, 실제 운동 기록 및 통계 분석 기능이 정상 작동하고 있다. 이는 CV(Computer Vision) 피벗 시 **앱 프론트엔드와 백엔드 인프라를 그대로 활용**할 수 있음을 의미하며, 신규 개발 대비 **약 4~6개월의 개발 기간 단축** 효과가 있다.

#### V2log 기(旣)구현 기능 목록

| 영역 | 구현 완료 기능 | 기술 스택 |
|------|--------------|----------|
| **인증/온보딩** | 이메일 회원가입/로그인, 사용자 프로필, 온보딩 플로우 | Supabase Auth |
| **운동 기록** | 세트별 무게/횟수 기록, 빠른 입력 UI(10초 기록), 세트 타입 구분(웜업/본세트/드롭/슈퍼세트) | Flutter + Riverpod |
| **루틴 관리** | 전문가 프리셋 루틴 5종, 루틴 라이브러리, 루틴→운동 세션 자동 생성 | Supabase DB |
| **통계 분석** | 주간/월간 볼륨 차트, 부위별 분석, 1RM 진행도(6개월 추이), 강도 존 분석 | fl_chart |
| **운동 라이브러리** | 운동 검색/필터, 시범 이미지, 상세 가이드(수행 방법·팁), 근육 맵 시각화 | muscle_selector 패키지 |
| **커뮤니티** | 오운완(오늘 운동 완료) 공유 카드 위젯, 이미지 저장/공유 | share_plus, gal |
| **부가 기능** | 휴식 타이머, 알림 설정(요일별 시간), 다크/라이트 테마 | flutter_local_notifications |

#### V2log 기술 아키텍처 현황

```
┌─────────────────────────────────────────────────────┐
│                    V2log App (Flutter)                │
├──────────┬──────────┬──────────┬────────────────────┤
│ 인증/온보딩│ 운동 기록  │ 통계 분석  │ 루틴 관리          │
│ (Auth)   │ (Workout)│ (Stats)  │ (Routine)          │
├──────────┴──────────┴──────────┴────────────────────┤
│         상태 관리: Riverpod 2.5+ (@riverpod)          │
│         라우팅: GoRouter (ShellRoute + 하단탭)         │
│         데이터 모델: Freezed + JsonSerializable        │
├─────────────────────────────────────────────────────┤
│              Backend: Supabase                       │
│   Auth │ PostgreSQL DB │ Storage │ Realtime          │
└─────────────────────────────────────────────────────┘
```

**핵심 포인트**: V2log의 데이터 모델(운동·세트·루틴·세션)과 백엔드 인프라는 CV 기반 자동 기록 시스템과 **100% 호환**된다. CV 엔진이 추출한 횟수·무게·운동 종류 데이터는 기존 `WorkoutSetModel`에 그대로 저장되며, 기존 통계·차트·1RM 계산(`FitnessCalculator`) 로직이 변경 없이 활용된다.

#### V2log 데이터베이스 스키마 (Supabase PostgreSQL)

이미 구축 완료된 9개 핵심 테이블은 CV 피벗 후에도 그대로 사용한다.

```
auth.users ← users(1:1) ← routines ← routine_exercises → exercises
                         ← workout_sessions ← workout_sets → exercises
                         ← exercise_records → exercises
                         ← body_records
preset_routines ← preset_routine_exercises → exercises
```

### (2) CV 기술 PoC(개념 검증) 준비 현황

센서 기반 접근에서 CV 기반으로 피벗하면서, 핵심 기술 요소에 대한 **사전 기술 검토 및 PoC 설계**를 완료하였다.

#### CV 핵심 기술 검토 결과

| 기술 요소 | 검토 라이브러리/모델 | 검토 결과 |
|----------|-------------------|----------|
| **포즈 추정(Pose Estimation)** | Google MediaPipe Pose | 33개 관절 키포인트, CPU에서 30+ FPS, 모바일 최적화 완료. 상용 수준 성능 확인 |
| **횟수 카운팅(Rep Counting)** | 관절 각도 기반 Rule Engine | 선행 연구에서 90~97% 정확도 입증. 관절 각도의 Peak/Valley 패턴으로 1회 반복 감지 |
| **폼 체크(Form Check)** | 관절 각도 임계값 비교 | 운동별 관절 각도 기준값 DB 구축 계획 수립 (대표자 16년 헬스 도메인 지식 기반) |
| **무게 감지 Phase 1** | 수동 입력 + 스마트 UX | 이전 기록 자동 팝업, 빠른 입력 UI는 V2log에 이미 구현 완료 |
| **무게 감지 Phase 2** | OCR(Tesseract/EasyOCR) + YOLOv8 | 핀머신 숫자판 OCR 인식률, 플레이트 객체 감지 정확도 벤치마크 설계 완료 |
| **운동 분류(Exercise Classification)** | LSTM + Rule-based Hybrid | 포즈 시퀀스 데이터로 운동 종류 자동 판별. 학술 논문 기반 구조 설계 완료 |
| **온디바이스 추론** | TensorFlow Lite / PyTorch Mobile | 개인정보 보호를 위한 온디바이스 처리 가능 확인. 영상 서버 전송 불필요 |

#### PoC 실행 계획

```
[PoC Phase 1: 4주]
├── MediaPipe Pose → Flutter 카메라 연동 (실시간 관절 좌표 추출)
├── 벤치프레스·스쿼트·데드리프트 3대 운동 횟수 카운팅 정확도 검증
└── 목표: 5회 이상 반복 시 카운팅 정확도 95% 이상

[PoC Phase 2: 4주]
├── 운동별 관절 각도 임계값 설정 (벤치프레스: 팔꿈치 각도 90° 이하 시 1회 인정)
├── 폼 체크 피드백 시나리오 검증 ("팔꿈치를 더 내리세요" 등)
└── 목표: 주요 폼 오류 3가지 이상 실시간 감지

[PoC Phase 3: 4주]
├── OCR 기반 핀머신 무게 숫자 인식 테스트
├── YOLO 기반 바벨 플레이트 감지 및 무게 추정 테스트
└── 목표: 통제된 환경에서 무게 인식 정확도 80% 이상
```

## 2-2. 창업아이템 실현 및 구체화/고도화 방안

### (1) 핵심 기술 구조: 센서에서 CV로의 전환

기존 센서 기반 접근은 **추가 하드웨어 비용, 체육관 설치 부담, 기기 유지보수** 등의 문제가 있었다. CV 기반 피벗을 통해 **스마트폰 카메라만으로** 동일한 측정 항목을 달성하며, 센서로는 불가능했던 **폼 체크 기능을 신규 추가**한다.

#### 센서 vs CV 기술 대체 비교표

| 측정 항목 | 기존 (센서 방식) | CV 피벗 후 | 장점 |
|----------|----------------|-----------|------|
| **횟수/반복 카운팅** | 가속도 센서 부착 | Pose Estimation 관절 각도 분석 | 센서 부착 불필요, 비접촉식 |
| **무게 감지 (핀머신)** | Hall 센서 설치 | OCR + 카메라 (숫자판 인식) | 기구 개조 불필요 |
| **무게 감지 (플레이트)** | RFID 태그 부착 | YOLO 객체 인식 (플레이트 감지) | 플레이트마다 태그 불필요 |
| **기구 인식** | NFC 태그 부착 | 카메라 위치 학습 / QR코드 | 체육관 인프라 변경 불필요 |
| **폼 체크** | **불가능** | 관절 각도 임계값 비교 **(신규!)** | 부상 예방 + 운동 효과 극대화 |
| **트래픽 관리** | 디바이스 감지 | 카메라 기구 사용 현황 감지 | 별도 센서 인프라 불필요 |

**비용 절감 효과**: 센서 방식은 기구당 센서 설치비 약 5~10만 원 + 유지보수 비용이 발생하지만, CV 방식은 사용자 스마트폰 또는 체육관 기(旣)설치 CCTV만으로 구현 가능하여 **초기 하드웨어 비용 0원**이다.

```
      │      │  │      │  │      │
      │      │  │      │  │      │
 90° ─┼──┐  ┌┼──┼┐  ┌──┼──┼┐  ┌─┼─      ← 팔꿈치 90° 굽힘
      │  │  ││  ││  │  │  ││  │ │
      │  └──┘│  │└──┘  │  │└──┘ │
      │      │  │      │  │     │
시간 ──┼──────┼──┼──────┼──┼─────┼──→
      1회     2회       3회

Peak(최고점)과 Valley(최저점)를 감지하여 1회 반복을 카운팅
정확도: 선행 연구 기준 90~97% (5회 이상 반복 시)
```

**관절 각도 계산 공식**:
```
θ = arccos( (BA⃗ · BC⃗) / (|BA⃗| × |BC⃗|) )

BA⃗ = 상완(어깨→팔꿈치) 벡터
BC⃗ = 전완(팔꿈치→손목) 벡터
θ  = 팔꿈치 각도
```

**B. 운동 자동 분류 (Exercise Classification)**

| 모델 | 입력 | 출력 | 방식 |
|------|-----|------|------|
| Rule Engine (1차) | 주요 관절 움직임 패턴 | 운동 대분류 (상체/하체/전신) | 규칙 기반: "무릎 각도 변화 > 30° → 하체 운동" |
| LSTM 모델 (2차) | 2초간 관절 좌표 시퀀스 (33×3×60프레임) | 운동 소분류 (벤치프레스, 스쿼트 등) | 시계열 딥러닝 |

**하이브리드 접근의 이점**: Rule Engine이 빠르게 대분류를 판단한 후, LSTM이 정밀 분류를 수행하므로 **속도와 정확도를 동시에 확보**한다.

**C. 폼 체크 (Form Check) - CV 피벗의 핵심 차별점**

센서 방식에서는 **원천적으로 불가능**했던 기능이 CV를 통해 가능해진다.

| 운동 | 체크 관절 | 정상 범위 | 폼 오류 판정 | 피드백 메시지 |
|------|----------|----------|-------------|-------------|
| 벤치프레스 | 팔꿈치 | 80°~100° | < 70° 또는 > 120° | "팔꿈치 각도를 조절하세요" |
| 스쿼트 | 무릎 | 70°~90° | 무릎이 발끝 앞으로 과도하게 나감 | "무릎이 발끝을 넘지 않게 하세요" |
| 데드리프트 | 허리(척추) | 170°~180° | < 160° (라운딩) | "허리를 펴세요! 부상 위험" |
| 오버헤드프레스 | 어깨+팔꿈치 | 팔꿈치 90°~180° | 허리 과신전 | "코어에 힘을 주세요" |

### (5) 기술 보호 전략

#### A. 데이터 해자 (Data Moat)

**운동별 관절 각도 기준값 DB**는 피카의 핵심 경쟁 우위이다.

| 보호 자산 | 내용 | 구축 방법 |
|----------|------|----------|
| **관절 각도 기준 DB** | 운동별 정상/비정상 관절 각도 범위, 폼 오류 유형, 교정 가이드 | 대표자 16년 헬스 트레이닝 경력 + 전문 트레이너 자문 |
| **사용자 운동 데이터** | 운동별 1RM 추이, 세트 기록, 폼 오류 패턴 | 서비스 운영 과정에서 자연 축적 (개인정보 비식별화 처리) |
| **기구별 학습 모델** | 체육관별 고정 카메라 위치에서의 기구 인식 모델 | B2B 파트너 체육관과 협력하여 현장 데이터 수집 |

이 데이터는 **16년간의 피트니스 도메인 전문 지식이 엔지니어링으로 코드화**된 것으로, 단순히 기술을 모방하는 것으로는 동일한 품질을 달성할 수 없다.

### (7) 개인정보 보호 설계 (Privacy by Design)

CV 기반 서비스에서 가장 중요한 **영상 데이터 개인정보 보호**를 설계 단계부터 반영한다.

```
[데이터 처리 흐름 - 개인정보 보호]

카메라 영상 → 온디바이스 AI 처리 → 관절 좌표(숫자)만 추출 → DB 저장
     │              │                    │
     │              │                    └── 개인 식별 불가능한 좌표값만 저장
     │              │
     │              └── 스마트폰 내에서만 처리, 서버 전송 없음
     │
     └── 영상은 즉시 폐기 (저장하지 않음)
```

| 원칙 | 구현 방법 |
|------|----------|
| **데이터 최소화** | 영상 원본 저장 안 함, 관절 좌표(33개 x,y,z 숫자)만 추출 |
| **온디바이스 처리** | TFLite/PyTorch Mobile로 스마트폰 내 추론, 서버 전송 없음 |
| **투명성** | 사용자에게 "영상은 저장되지 않습니다" 명확히 고지 |
| **선택권** | 카메라 기능 미사용 시 기존 수동 입력 모드 100% 지원 |
